# Day 1: 建立 RAG 系统的正确心智模型

## 今日定位

今天不学如何启动 `MODULAR-RAG-SERVER`, 也不学如何更换模型或修改配置。

今天只建立一个最重要的能力: 看到一个所谓的“RAG 项目”时, 能准确判断它包含什么、缺少什么、错误可能发生在哪一层。

这个能力决定了后续阅读源码时, 你是在记住一堆类名, 还是真正理解一个 AI 系统。

## 今日学习目标

学完今天的内容后, 你应该能够:

1. 用自己的话说清楚 RAG 为什么出现。
2. 区分参数知识与外部知识。
3. 区分 Index-time 和 Query-time。
4. 区分文档处理、召回、融合、重排、生成与引用。
5. 判断一次错误属于知识库问题、检索问题还是生成问题。
6. 根据源码证据判断当前案例的真实能力边界。

---

## 第一部分: RAG 为什么出现

### 1. 大模型的回答从哪里来

大模型的核心能力是: 根据当前上下文, 预测下一个最可能出现的 Token。

它的回答主要依赖两类信息:

- 训练过程中压缩到模型参数里的统计规律;
- 当前请求中提供的上下文。

模型参数中的知识可以理解为一种“被压缩的统计性记忆”。它不是一个可以精确查询的数据库, 也不能保证每条信息都可以追溯到原始来源。

### 2. 只依赖模型参数会遇到什么问题

#### 知识不新

模型训练完成后, 现实世界仍在变化。新发布的制度、新版接口和企业内部文档并不会自动进入模型参数。

#### 私有知识不可见

公司的业务文档、产品手册、事故记录和客户数据通常没有参与公开模型的训练。

#### 来源无法追溯

即使模型给出了正确答案, 使用者也可能无法确认它依据的是哪一份文档、哪一页或哪一段。

#### 容易生成“语言流畅但事实错误”的内容

模型的优化目标是生成概率上连贯的文本, 不是自动证明每一句话的事实性。

### 3. RAG 的基本思路

RAG 的全称是 Retrieval-Augmented Generation, 即检索增强生成。

它不是让模型“永久记住”新知识, 而是在回答当前问题前, 先从外部知识源中找到相关材料, 再把这些材料放进当前上下文。

```text
用户问题
   ↓
检索相关材料
   ↓
组装“问题 + 检索材料 + 回答约束”
   ↓
LLM 生成回答
   ↓
返回答案与引用
```

这里有一个必须记住的结论:

> RAG 不保证模型永远正确。RAG 只是把可靠外部证据送到模型面前, 降低模型凭空回答的概率。

## 第二部分: 参数知识与外部知识

### 1. 参数知识

参数知识存在于模型权重中。

特点:

- 调用方便, 回答时不需要额外检索;
- 适合通用语言规律和常识;
- 难以快速更新;
- 难以精确追溯;
- 不适合动态私有数据。

### 2. 外部知识

外部知识存在于模型之外, 例如:

- PDF 文档;
- 企业 Wiki;
- 数据库;
- 工单系统;
- 代码仓库;
- 搜索引擎;
- 实时 API。

特点:

- 可以独立更新;
- 可以进行权限控制;
- 可以追溯原文;
- 需要额外的索引、检索和数据治理;
- 检索错误会直接影响最终回答。

### 3. RAG 不等于微调

| 维度 | RAG | 微调 |
|---|---|---|
| 主要目的 | 向当前请求提供外部证据 | 改变模型的行为模式或任务能力 |
| 知识位置 | 模型外部 | 模型参数内 |
| 更新速度 | 更新文档或索引即可 | 需要重新训练与验证 |
| 来源引用 | 天然适合 | 通常难以精确追溯 |
| 适合场景 | 私有、动态、需引用的知识 | 格式、风格、特定行为或任务适配 |

一个系统可以同时使用 RAG 和微调, 但两者解决的不是同一类问题。

## 第三部分: 完整 RAG 系统的两条链路

RAG 系统不是一条单一流程, 而是两条在不同时间运行的链路。

## 1. Index-time: 知识准备链

Index-time 也可以称为摄取、建库或索引构建阶段。

```text
原始文件
→ 加载与解析
→ 统一文档表示
→ 分块
→ 清洗与元数据增强
→ Embedding / 词项统计
→ 写入向量索引和倒排索引
```

这条链的核心目标不是“保存文件”, 而是把文件加工成对查询友好的数据。

它回答的问题是:

> 为了让未来的查询快速找到正确材料, 今天应该如何处理这份文档?

## 2. Query-time: 在线回答链

```text
用户问题
→ 查询处理
→ 候选召回
→ 多路结果融合
→ 重排
→ 选择最终上下文
→ 组装 Prompt
→ LLM 生成
→ 引用和结构化输出
```

这条链回答的问题是:

> 面对当前问题, 系统应该找出哪些证据, 并如何约束模型使用这些证据?

## 3. 两条链的关系

Query-time 的上限往往在 Index-time 就已经被决定了。

例如:

- PDF 解析时丢失表格, 查询时无法召回表格内容;
- 分块把一个完整规则切成两半, 检索到任何一半都不够;
- 元数据没有保留文档来源, 最终无法生成可信引用;
- 摄取时使用了一个 Embedding 模型, 查询时使用另一个不兼容模型, 向量距离就失去意义。

所以诊断 RAG 时不能只盯着最终回答。

## 第四部分: 六个必须分开的技术阶段

## 1. 文档处理

职责:

- 读取 PDF、Markdown 等数据源;
- 恢复文本、图片和结构;
- 分块;
- 保留来源和页码;
- 生成检索需要的索引。

该阶段失败时, 知识可能根本没有进入检索系统。

## 2. 召回 Retrieval

职责:

- 从大量 Chunk 中快速选出候选集;
- 尽可能不漏掉相关内容。

召回阶段优先关注 Recall, 也就是“正确答案是否进入候选集”。

## 3. 融合 Fusion

职责:

- 合并多个召回器的候选列表;
- 处理不同检索器分数不可直接比较的问题;
- 利用多路共识提升候选排名。

本案例使用 RRF 合并 Dense 和 Sparse 排名。

## 4. 重排 Rerank

职责:

- 对较小的候选集进行更精确但更昂贵的判断;
- 把真正能回答当前问题的材料排在前面。

召回解决“不要漏”, 重排解决“前面尽量准”。

## 5. 生成 Generation

职责:

- 把查询、检索证据和行为约束组装成 Prompt;
- 调用 LLM;
- 要求模型基于证据回答;
- 在证据不足时拒绝猜测。

注意: 检索到正确文档不代表模型一定会忠实使用它。

## 6. 引用 Citation

职责:

- 将回答与具体证据建立关联;
- 返回文档、页码、Chunk ID 和摘要;
- 支持人工复核和系统审计。

引用存在不代表回答必然受引用支持。还需要检查回答中的每个事实主张是否真的能从引用文本中推导出来。

## 第五部分: 用分层方法诊断 RAG 错误

不要把所有错误都称为“大模型幻觉”。

先问五个问题:

1. 正确知识是否已经进入知识库?
2. 正确知识是否被分成了可独立理解的 Chunk?
3. 正确 Chunk 是否进入召回候选集?
4. 正确 Chunk 是否在融合和重排后保留在最终上下文中?
5. 模型是否忠实使用了这些上下文?

### 场景 A: 答案所在文档没有被摄取

问题层级: 知识准备。

此时更换 Prompt 没有意义, 因为检索系统中根本没有正确证据。

### 场景 B: 正确 Chunk 存在, 但 Top-K 中没有它

问题层级: 召回。

检查方向:

- Query 表达与文档表达是否差异过大;
- Embedding 是否匹配当前语言和领域;
- BM25 Token 策略是否正确;
- Top-K 是否过小;
- 元数据过滤是否误排除了结果。

### 场景 C: 正确 Chunk 在召回集中, 但被排到后面

问题层级: 融合或重排。

检查方向:

- 融合策略是否过度偏向某一路;
- Reranker 是否真的理解查询意图;
- 候选文本是否因分块而失去必要上下文。

### 场景 D: 最终上下文正确, 但回答仍然错误

问题层级: 生成。

检查方向:

- Prompt 是否明确要求基于证据;
- 多份材料是否相互冲突;
- 上下文是否过长;
- 模型是否增加了证据中不存在的细节;
- 是否缺少“证据不足时明确说不知道”的约束。

### 场景 E: 回答正确, 但引用指向无关内容

问题层级: 引用对齐。

这类错误会给人一种“有引用就可信”的假象, 实际上风险很高。

## 第六部分: 项目案例分析

## 1. 统一数据契约

阅读文件:

```text
src/core/types.py
```

重点观察:

- `Document`: 代表解析后、未分块的文档;
- `Chunk`: 代表可独立索引的文本片段;
- `ChunkRecord`: 代表已带 Dense/Sparse 表示、可写入存储的记录;
- `ProcessedQuery`: 代表处理后的查询;
- `RetrievalResult`: 代表统一的检索结果。

这些类不只是为了写类型提示。它们定义了每个技术阶段的边界。

## 2. Index-time 的项目实现

阅读文件:

```text
src/ingestion/pipeline.py
```

项目的主要摄取链:

```text
文件完整性检查
→ 文档加载
→ 文档分块
→ Chunk 清洗和元数据增强
→ Dense + Sparse 编码
→ Chroma + BM25 + 图片索引写入
```

源码中的六个 Stage 说明作者已经把“文件进库”看成一个可观测的数据流水线, 而不是一个单独函数。

## 3. Query-time 的项目实现

阅读文件:

```text
src/mcp_server/tools/query_knowledge_hub.py
src/core/query_engine/hybrid_search.py
src/core/query_engine/reranker.py
src/core/response/response_builder.py
```

实际链路:

```text
MCP Tool 参数校验
→ 初始化查询组件
→ Dense + Sparse 召回
→ RRF 融合
→ 可选 Rerank
→ ResponseBuilder
→ 检索片段 + 引用 + 可选图片
```

## 4. 当前项目的真实边界

这是今天最重要的源码结论。

`QueryKnowledgeHubTool.execute` 完成检索和重排后, 直接把 `RetrievalResult` 交给 `ResponseBuilder.build`。

`ResponseBuilder` 的主要工作是:

- 遍历检索结果;
- 截取文本片段;
- 显示相关度;
- 生成来源引用;
- 组装 MCP 内容块。

主查询链中没有看到以下完整闭环:

```text
检索上下文
→ 组装问答 Prompt
→ 调用 LLM 生成综合答案
→ 检查答案是否受证据支持
```

因此对该项目最准确的定位是:

> 它是一个模块化、可观测、可通过 MCP 调用的 RAG 检索基础设施案例, 而不是已经完成所有生成与忠实性校验的完整 RAG 问答系统。

这不是对项目的贬低, 而是源码阅读必须具备的能力: 用实际调用关系界定系统能力, 而不是只根据仓库名称和 README 下结论。

## 第七部分: 通用工程经验

## 经验 1: 先定义系统边界, 再谈优化

如果一个系统只返回检索片段, 就不应该用“答案忠实性”来描述它的主能力。反过来, 如果一个系统生成了最终答案, 就不能只评估检索命中率。

## 经验 2: 不要用最终结果掩盖中间过程

优秀的 AI 系统会保留中间信息:

- 文档如何被解析;
- 生成了哪些 Chunk;
- Dense 召回了什么;
- Sparse 召回了什么;
- 融合和重排后发生了什么变化;
- 哪些材料最终进入模型上下文。

没有中间过程, 系统只能“猜测性调参”。

## 经验 3: 检索质量和生成质量必须分开评估

检索评估关心:

- 正确 Chunk 是否被召回;
- 排名是否靠前;
- 候选是否有足够多样性。

生成评估关心:

- 答案是否回答了问题;
- 答案是否受检索证据支持;
- 是否添加了无证据事实;
- 引用是否与具体主张对齐。

## 经验 4: 用数据流阅读项目, 不用目录名阅读项目

看一个系统时, 优先追踪这些问题:

1. 输入是什么类型?
2. 经过哪些变换?
3. 每次变换丢失或增加了什么信息?
4. 最终输出是什么?
5. 输出是否真的完成了系统宣称的任务?

## 第九部分: 参考答案

建议先独立完成练习, 再阅读本节。

## 练习 1 参考结构

```text
Index-time
原始文档
→ 解析后 Document
→ Chunk[]
→ 向量/词项统计
→ 可检索索引

Query-time
用户查询
→ 查询表示
→ 候选 Chunk[]
→ 融合和重排后的 Context[]
→ Prompt
→ LLM Answer
→ Answer + Citations
```

## 练习 2 参考答案

- 问题 A: 先查摄取状态、版本元数据和过滤规则, 再查召回。
- 问题 B: 先查 Sparse/BM25 精确词项召回, 不应只依赖 Dense。
- 问题 C: 属于生成忠实性问题, 检查 Prompt、模型回答和事实校验。
- 问题 D: 属于引用对齐问题。

## 练习 3 参考答案

当前主链路完成了文档摄取、Dense/Sparse 召回、RRF 融合、可选重排和引用封装。检索结果会直接交给 `ResponseBuilder` 格式化, 未形成“检索上下文 → LLM 生成答案 → 忠实性校验”的完整闭环, 因此更准确地说它是 RAG 检索基础设施。

## 练习 4 参考方向

### 元数据

至少包含:

- 制度名称;
- 制度版本;
- 生效日期;
- 失效日期或当前状态;
- 发布部门;
- 原始文档路径;
- 页码或章节;
- 文档内容哈希。

### 版本控制

查询时使用元数据过滤, 默认只召回当前生效版本。如果用户查询历史规则, 才显式指定时间或版本。

### 引用

至少返回制度名称、版本、生效日期、来源路径、页码/章节和支持当前结论的文本摘要。

### 证据不足

明确告知“当前知识库中没有找到生效制度作为依据”, 不允许模型根据通用常识补齐公司规则。
