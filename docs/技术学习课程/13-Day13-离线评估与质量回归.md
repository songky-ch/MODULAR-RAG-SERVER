# Day 13: 离线评估与质量回归

## 今日定位

检索系统的修改不能靠“看起来不错”验收。离线评估用固定查询集和可重复指标回答: 新分块、新模型或新参数到底提升还是退化。

## 学习目标

- 设计最小 Golden Test Set;
- 掌握 Hit Rate@K、Recall@K 和 MRR;
- 区分检索评估与生成评估;
- 能运行参数对照实验并解释结果。

## 1. 技术背景

更换 Embedding、调整 Chunk size 或 RRF 权重, 可能提升某类查询却伤害另一类。只用几个演示问题容易受到选择偏差影响。质量工程需要:

```text
固定数据集 + 固定指标 + 固定运行条件 + 变更前后对比
```

这就是离线回归评估。

## 2. 核心原理

### 2.1 Golden Test Set

每条测试至少包含:

```text
query
expected_chunk_ids
reference_answer
category
```

`expected_chunk_ids` 用于检索指标, `reference_answer` 用于生成指标, `category` 用于分组分析。Ground Truth 应经过人工核验。

### 2.2 Hit Rate@K

Top-K 中只要出现任意一个相关项就记为 1:

$$
HitRate@K=\frac{1}{N}\sum_{i=1}^{N}
\mathbb{1}(\text{Top-K 命中})
$$

适合“找到一个正确证据即可”的场景。

### 2.3 Recall@K

$$
Recall@K=
\frac{|\text{Top-K}\cap\text{Relevant}|}
{|\text{Relevant}|}
$$

适合一个问题需要多个证据的场景。

### 2.4 MRR

对每个查询取第一个相关项排名的倒数:

$$
MRR=\frac{1}{N}\sum_{i=1}^{N}
\frac{1}{rank_i}
$$

未命中贡献 0。MRR 不只关心是否命中, 也关心首次命中的位置。

### 2.5 检索与生成分层

- 检索指标: ID 命中、排名和覆盖;
- 生成指标: 答案忠实性、相关性、完整性;
- 端到端指标: 用户任务是否完成。

检索没找到证据时, 生成评估再高也可能是模型凭内部知识回答。

## 3. 项目案例分析

### 3.1 源码阅读顺序

1. `src/observability/evaluation/eval_runner.py`;
2. `src/libs/evaluator/base_evaluator.py`;
3. `src/libs/evaluator/custom_evaluator.py`;
4. `src/observability/evaluation/composite_evaluator.py`;
5. `src/observability/evaluation/ragas_evaluator.py`。

### 3.2 数据契约

`GoldenTestCase` 保存 query、expected answer 和 expected Chunk IDs。`QueryResult` 保存实际返回 ID、生成答案、指标和错误。`EvalReport` 聚合单条结果与总体指标。

### 3.3 EvalRunner 主链

```text
load_test_set
  → EvalRunner.run
  → _evaluate_single
      → _retrieve
      → 提取 retrieved_chunk_ids
      → _generate_answer
      → evaluator.evaluate
  → _aggregate_metrics
  → EvalReport
```

`CustomEvaluator` 实现 Hit Rate 和 MRR, 并兼容 `id`、`chunk_id`、`document_id`、`doc_id` 等字段。字段兼容提高接入性, 但 Golden Set 最好仍统一使用稳定 Chunk ID。

### 3.4 Composite 与 Ragas

`CompositeEvaluator` 聚合多个评估器, 允许同时得到确定性检索指标和模型型指标。`RagasEvaluator` 面向答案质量评价, 可能依赖外部模型, 因而成本、波动和失败率都高于纯 ID 指标。

### 3.5 项目边界

当前查询主链主要返回检索片段, 并非完整答案生成链。`EvalRunner` 中的答案生成逻辑不能被误读为项目已经具备成熟生成质量评估。学习时应优先把检索指标做扎实。

## 4. 通用工程经验

- Golden Set 必须覆盖语义、关键词、编号、多证据和无答案查询;
- 报告总体指标也要按查询类别分组;
- 固定语料版本、模型版本和参数;
- 先用确定性指标建立回归门槛, 再引入 LLM Judge;
- 保存每条失败样本, 指标下降后才能具体诊断。

## 5. 实践练习

建立 10 条 Golden Query:

- 3 条同义改写;
- 2 条精确术语或编号;
- 2 条多证据问题;
- 2 条易混淆问题;
- 1 条无答案问题。

比较两组配置:

```text
A: Dense only, top_k=5
B: Dense + BM25 + RRF, top_k=5
```

输出:

| 配置 | Hit Rate@5 | MRR | 平均延迟 |
|---|---:|---:|---:|
| A |  |  |  |
| B |  |  |  |

逐条查看排名发生变化的查询, 给出“为什么提升或退化”的证据, 不只汇报平均值。

## 6. 验收问题

1. Hit Rate 与 Recall 有什么区别?
2. 为什么 MRR 能反映首个正确结果的位置?
3. Golden Set 为什么需要查询类别?
4. 检索评估和生成评估为什么必须分层?
5. LLM Judge 为什么不能成为唯一回归标准?

## 7. 今日学习成果

完成本课后, 你应能构建最小可信的检索评估闭环, 用数据比较参数和算法变更, 并定位平均指标背后的具体样本变化。
