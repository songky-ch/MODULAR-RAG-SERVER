# Day 6: 混合检索与多路召回

## 今日定位

Dense 擅长语义改写, BM25 擅长精确词项。混合检索不是简单地“多跑一个算法”, 而是用互补召回路径扩大候选覆盖面, 再把结果交给统一融合层。

## 学习目标

- 区分召回率、精确率和候选预算;
- 理解多路召回的互补性;
- 掌握并行扇出、失败隔离和结果汇合;
- 能为不同检索器设计 Top-K 配额。

## 1. 技术背景

单一召回器必然有系统性盲区:

- Dense 可能错过编号、缩写和罕见专名;
- Sparse 可能错过同义改写和概念性提问。

生产检索通常先让多个低成本召回器各自产生候选, 再进行融合和精排。这是典型的 Candidate Generation 架构。

## 2. 核心原理

### 2.1 召回阶段的目标

召回阶段首先追求“相关答案是否进入候选集”, 而不是最终顺序完美。可用 Recall@K 表示:

$$
Recall@K=\frac{\text{Top-K 中命中的相关项数}}{\text{全部相关项数}}
$$

候选越多, 召回通常越高, 但融合、Rerank 和下游上下文成本也越高。

### 2.2 并行扇出

```text
                 ┌→ Dense Retriever ─┐
Query Processing ┤                    ├→ Fusion
                 └→ Sparse Retriever ┘
```

若两路耗时分别是 $T_d$ 和 $T_s$, 串行耗时约为 $T_d+T_s$, 并行理想耗时约为 $\max(T_d,T_s)$。实际还要加线程调度与汇合成本。

### 2.3 候选预算

最终要返回 5 条结果时, 每路不应只取 5 条。相关项可能在单路排名靠后。常见做法是每路召回 $m\times K$ 个候选, 融合后再截断。$m$ 应通过评估确定。

### 2.4 失败语义

多路召回应区分:

- 一路成功、一路失败: 返回成功路径的部分结果并标记降级;
- 两路都成功但某路无结果: 正常业务结果;
- 两路都失败: 整体检索失败。

“空结果”和“异常”不能混为一谈。

## 3. 项目案例分析

### 3.1 源码阅读顺序

1. `src/core/query_engine/hybrid_search.py`;
2. `src/core/query_engine/dense_retriever.py`;
3. `src/core/query_engine/sparse_retriever.py`;
4. `src/core/query_engine/query_processor.py`;
5. `src/core/query_engine/fusion.py`。

### 3.2 主调用关系

```text
HybridSearch.search
  → _process_query
  → _merge_filters
  → _run_retrievals
      → _run_parallel_retrievals
          → _run_dense_retrieval
          → _run_sparse_retrieval
  → _fuse_results
  → metadata post-filter
  → HybridSearchResult
```

`_run_parallel_retrievals()` 使用 `ThreadPoolExecutor(max_workers=2)` 并行执行 Dense 与 Sparse。两路都可用时进入 RRF; 只有一路可用时使用该路结果并设置 `used_fallback=True`。

### 3.3 Query Processing 的意义

查询进入召回器前先被规范化并抽取关键词。Dense 路径偏向使用原始查询保留自然语义, Sparse 路径使用关键词以提高词项匹配。统一的 `ProcessedQuery` 是多路召回的数据契约。

### 3.4 元数据过滤

过滤可以发生在召回前或融合后:

- 前置过滤减少候选规模, 但依赖后端支持;
- 后置过滤兼容性高, 但可能把 Top-K 全部过滤掉。

项目保留 post-filter fallback, 体现了后端能力不一致时的兼容思路。

## 4. 通用工程经验

- 并行执行前先确认依赖对象是否线程安全;
- 为每一路分别记录耗时、候选数和异常;
- 召回预算应由离线评估和延迟预算共同决定;
- 不要直接相加 Dense 与 BM25 原始分数;
- 降级结果要显式携带状态, 不能伪装成完整成功。

## 5. 实践练习

实现两个检索器:

- `SemanticRetriever`: 用自定义同义词表计算语义匹配;
- `KeywordRetriever`: 用 Token 重合数排序。

使用线程池并行执行, 汇总时保留:

```text
source
rank
raw_score
latency_ms
error
```

设计三类查询: 同义改写、精确编号、两者混合。分别比较只用一条路径和双路召回时, 正确答案是否进入 Top-10。

再模拟一路抛出异常, 验证系统返回另一路结果并明确标记降级。

## 6. 验收问题

1. 为什么召回阶段不追求最终排序完美?
2. 并行召回何时不会降低延迟?
3. 为什么每路候选数通常大于最终 Top-K?
4. 前置过滤和后置过滤各有什么风险?
5. 空结果与检索异常的语义差异是什么?
