# Day 5: BM25 与稀疏检索

## 今日定位

BM25 是经典关键词相关性算法。它不理解深层语义, 但对编号、术语、人名和原词匹配非常可靠。本课从倒排索引到项目实现完整拆解稀疏检索。

## 学习目标

- 理解 Token、TF、DF、IDF 和倒排索引;
- 能解释并手算简化 BM25;
- 理解中文 2-gram 的用途和代价;
- 掌握索引时统计与查询时打分的分工。

## 1. 技术背景

如果每次查询都扫描全部文档, 计算成本与语料规模线性增长。倒排索引把关系反转:

```text
文档 → 包含哪些词
```

变为:

```text
词 → 出现在哪些文档
```

查询只访问相关词的 posting list, 无需遍历全库。

## 2. 核心原理

### 2.1 基础统计

- TF: 词在当前文档中出现次数;
- DF: 包含该词的文档数量;
- IDF: 词区分文档的能力。越常见的词权重越低;
- document length: 文档 Token 数量;
- average document length: 语料平均长度。

项目的 IDF 和 BM25 核心思想可以写成:

$$
\operatorname{score}(q,d)=
\sum_{t\in q}\operatorname{IDF}(t)
\frac{tf(t,d)(k_1+1)}
{tf(t,d)+k_1(1-b+b\frac{|d|}{avgdl})}
$$

### 2.2 参数含义

- $k_1$: 控制 TF 饱和。一个词重复很多次时, 得分不会无限线性增加;
- $b$: 控制文档长度归一化。$b=0$ 不惩罚长文档, $b=1$ 完全应用长度校正。

项目默认 `k1=1.5`, `b=0.75`。

### 2.3 中文 Tokenization

英文可以按空格近似分词, 中文没有天然空格。项目的稀疏编码会使用字符级与 2-gram 特征。比如:

```text
知识检索 → 知, 识, 检, 索, 知识, 识检, 检索
```

2-gram 可以增强短语匹配, 但会扩大词表并引入部分无意义组合。

## 3. 项目案例分析

### 3.1 源码阅读顺序

1. `src/ingestion/embedding/sparse_encoder.py`: 文本 Token 化与词频统计;
2. `src/ingestion/storage/bm25_indexer.py`: 建索引、保存、加载、查询;
3. `src/core/query_engine/query_processor.py`: 查询侧 Token 化;
4. `src/core/query_engine/sparse_retriever.py`: 把查询结果还原为统一检索结果。

### 3.2 索引结构

`SparseEncoder.encode()` 为每个 Chunk 产生:

```text
chunk_id
term_frequencies
doc_length
unique_terms
```

`BM25Indexer.build()` 再聚合为:

```text
term
  → document_frequency
  → postings[{chunk_id, tf, doc_length}]
```

查询时, `BM25Indexer.query()` 只遍历查询 Token 对应的 postings, 调用 `_calculate_bm25_score()` 累加得分, 最后排序。

### 3.3 调用关系

```text
索引:
Chunk → SparseEncoder → term stats → BM25Indexer.build → 本地索引文件

查询:
query → QueryProcessor.tokenize → BM25Indexer.query
      → chunk_id + score → SparseRetriever 补齐文本与元数据
```

### 3.4 一致性要求

索引时与查询时必须采用相同 Token 规则。若索引使用 2-gram, 查询只按单字处理, 两侧词项空间不一致, 相关文档也无法命中。

## 4. 通用工程经验

- 分词策略本身就是模型, 变化后应重建稀疏索引;
- 停用词策略要由语料和查询验证, 不要机械删除;
- 保存语料统计时应带版本, 防止加载旧格式;
- 精确术语和编号优先利用 BM25;
- BM25 分数只能在同一次查询和同一索引内合理比较, 不宜直接与向量分数相加。

## 5. 实践练习

为五篇短文实现简化 BM25:

1. Token 化所有文档;
2. 构建倒排表和 DF;
3. 计算 `avgdl`;
4. 对查询逐词累加 BM25;
5. 返回 Top-3。

至少设计三类查询:

- 精确编号: `GB/T 35273`;
- 中文短语: `知识检索`;
- 同义改写: 文档和查询字面不同。

分别使用单字、2-gram、单字加 2-gram, 比较结果。记录哪种策略提高了命中, 哪种制造了噪声。

## 6. 验收问题

1. 倒排索引为什么能避免全文扫描?
2. IDF 为什么降低高频词权重?
3. $k_1$ 和 $b$ 分别控制什么?
4. 为什么索引和查询必须共用 Token 规则?
5. BM25 在哪些查询上通常优于 Dense Retrieval?
