# Day 3: 文档分块与上下文工程

## 今日定位

分块不是“把长文本切短”, 而是在检索成本、语义完整性和答案可引用性之间设计信息边界。本课研究如何把原始文档变成适合召回的最小知识单元。

## 学习目标

- 理解 Chunk size、overlap、结构边界的相互制约;
- 区分固定长度、递归分块、结构感知分块;
- 理解稳定 Chunk ID 和继承元数据为何属于分块职责;
- 能根据文档结构设计并验证分块策略。

## 1. 技术背景

Embedding 模型和大模型都有上下文长度限制。直接把整篇文档作为检索单位会产生两个问题:

1. 文档向量混合多个主题, 查询与局部事实的相似度被稀释;
2. 即使召回成功, 返回内容过长, 后续排序和生成成本都很高。

但切得过碎也会丢失标题、限定条件和跨句关系。因此分块的本质是确定“一个检索结果至少要保留多少上下文”。

## 2. 核心原理

### 2.1 三个核心变量

- `chunk_size`: 单块允许的最大长度。越大, 上下文越完整, 但主题越容易混杂;
- `chunk_overlap`: 相邻块重复的内容。可以保留跨边界信息, 但会增加索引量和重复召回;
- separator hierarchy: 优先在段落、换行、句子等自然边界处切分。

递归分块通常按照“段落 → 换行 → 句子 → 字符”的顺序尝试分隔符。当前层级无法满足大小限制时, 才进入更细粒度。

### 2.2 结构感知分块

财报、合同、手册等文档不是普通连续文本。标题定义后续段落语境, 表格又要求表头和数据行共同出现。结构感知策略会:

- 识别章节标题并维护层级栈;
- 将章节路径作为上下文前缀;
- 识别表格边界, 避免表头与数据分离;
- 超长表格拆分时重复必要的标题和表头。

### 2.3 Chunk 身份与血缘

一个 Chunk 至少需要:

- 稳定身份: 可以用于 Upsert、去重、评估和引用;
- 父文档身份: 能追溯来源;
- 顺序: 能恢复原文邻接关系;
- 元数据: 文件名、页码、章节等过滤条件。

项目中的 Chunk ID 形式为:

```text
{document_id}_{chunk_index:04d}_{text_sha256前8位}
```

它同时编码父文档、位置和内容变化。文本变化会改变 ID, 相同输入则得到相同 ID。

## 3. 项目案例分析

### 3.1 源码阅读顺序

1. `src/libs/splitter/base_splitter.py`: 分块器抽象契约;
2. `src/libs/splitter/recursive_splitter.py`: 通用递归策略;
3. `src/libs/splitter/financial_report_splitter.py`: 标题和表格感知策略;
4. `src/libs/splitter/splitter_factory.py`: 根据配置选择实现;
5. `src/ingestion/chunking/document_chunker.py`: 将文本片段转换成领域 `Chunk`。

### 3.2 调用关系

```text
Document
  → DocumentChunker.split_document
  → SplitterFactory 创建具体 Splitter
  → splitter.split_text
  → 逐片生成稳定 chunk_id
  → 继承文档元数据并补充 chunk_index
  → List[Chunk]
```

### 3.3 关键实现

`DocumentChunker.split_document()` 不亲自实现所有切分算法, 而是负责领域层编排。`RecursiveSplitter.split_text()` 处理通用文本; `FinancialReportSplitter` 维护章节上下文, 并对表格采用专门逻辑。

这种设计分离了两个变化:

- “如何切文本”由 Splitter 负责;
- “如何把切分结果变成可追踪 Chunk”由 DocumentChunker 负责。

### 3.4 实现边界

当前实现属于单层 Chunking。它没有建立 Parent Chunk 与 Child Chunk 两套索引, 因而不能直接实现“小块召回、大块返回”。课程中要理解该高级模式, 但不要把它误认为仓库已经实现。

## 4. 通用工程经验

- 不要只用平均字符数评价分块, 要用真实查询评估召回;
- overlap 不是越大越好, 重复块会挤占 Top-K;
- 结构化文档优先保留标题路径和表头;
- Chunk ID 必须确定性生成, 不应每次随机创建;
- 分块参数变化意味着索引语义发生变化, 应视为需要重建索引。

## 5. 实践练习

准备一篇包含标题、三个段落和一张表格的 800 至 1200 字文本, 实现三种策略:

1. 每 200 字硬切, overlap 为 0;
2. 每 200 字递归切, overlap 为 30;
3. 标题感知切分, 每块携带章节路径。

设计五个查询, 对每种策略记录:

| 指标 | 含义 |
|---|---|
| 首个相关块排名 | 是否容易召回 |
| 事实是否完整 | 条件和结论是否在同一块 |
| 重复块数量 | overlap 的代价 |
| 无关文本比例 | 块是否过大 |

不要接入向量数据库。可以先用关键词重合数作为简易检索分数, 实验重点是信息边界。

## 6. 验收问题

1. 为什么 Chunk 越小不一定召回越准?
2. overlap 解决了什么问题, 又制造了什么问题?
3. 为什么财务表格不能按普通段落切分?
4. Chunk ID 中加入内容哈希有什么价值?
5. 什么时候应采用 Parent-Child Chunking?

## 7. 今日学习成果

完成本课后, 你应能够从文档结构、查询粒度和下游成本三个维度设计分块策略, 并能用实验结果而不是直觉选择参数。
