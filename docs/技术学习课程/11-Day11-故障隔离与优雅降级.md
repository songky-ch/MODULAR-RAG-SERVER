# Day 11: 故障隔离与优雅降级

## 今日定位

优雅降级不是吞掉异常, 而是在部分能力失效时返回语义明确的有限结果, 同时保留可诊断信息。本课研究检索、精排和评估中的故障边界。

## 学习目标

- 区分 Fail-fast、Fail-soft 和静默失败;
- 理解核心能力与增强能力的不同故障策略;
- 掌握部分结果、回退状态和聚合错误;
- 能为并行依赖设计清晰的失败矩阵。

## 1. 技术背景

RAG 查询涉及多个外部或可选组件:

- Embedding API;
- 向量数据库;
- BM25 本地索引;
- Reranker 服务;
- 评估模型。

任何组件都可能超时或抛错。全部 Fail-fast 会降低可用性, 全部捕获又会让错误被隐藏。正确策略取决于该能力是否是当前请求的必要条件。

## 2. 核心原理

### 2.1 三种语义

- Fail-fast: 必要前置条件不成立, 立即失败;
- Fail-soft: 增强能力失败, 返回基础能力结果并标记降级;
- Silent failure: 吞掉异常且不暴露状态, 应避免。

### 2.2 失败矩阵

以 Dense 和 Sparse 双路召回为例:

| Dense | Sparse | 结果 |
|---|---|---|
| 成功 | 成功 | 融合 |
| 成功 | 失败 | Dense 降级结果 |
| 失败 | 成功 | Sparse 降级结果 |
| 失败 | 失败 | 整体失败 |

“成功但无命中”仍是成功, 不能当作异常路径。

### 2.3 降级状态模型

返回对象至少应表达:

```text
data
used_fallback
fallback_reason
failed_component
```

这样调用方、监控和用户界面可以区分完整结果与有限结果。

### 2.4 超时与取消

异常捕获只处理已经结束的失败。远程调用若无限等待, 系统仍不可用。可靠边界还需要超时、资源隔离和取消传播。项目展示了回退语义, 但完整的超时与熔断治理仍是进一步的生产实践。

## 3. 项目案例分析

### 3.1 HybridSearch

阅读 `src/core/query_engine/hybrid_search.py`:

- 两路召回并行执行;
- 单路失败时保留另一条路径;
- `HybridSearchResult.used_fallback` 暴露降级;
- 无 Fusion 时交错合并是算法级回退;
- Dense 和 Sparse 都不可用时不能伪造成功结果。

### 3.2 CoreReranker

阅读 `src/core/query_engine/reranker.py`:

- Rerank 是检索后的增强能力;
- 后端异常时按配置返回原排序;
- 元数据加入 `rerank_fallback=True`;
- `RerankResult` 保存 `fallback_reason`;
- 禁用回退时抛出 `RerankError`。

### 3.3 CompositeEvaluator

阅读 `src/observability/evaluation/composite_evaluator.py`:

- 多个 Evaluator 顺序运行并聚合指标;
- 单个评估器失败不必使其他指标丢失;
- 评估属于离线增强能力, 通常适合部分成功;
- 失败必须进入错误信息或日志。

### 3.4 变换阶段

元数据增强、图像描述等处理也涉及增强能力。设计时要先回答: 没有增强结果时, 原始 Chunk 是否仍具备业务价值? 如果是, 可以回退原内容; 如果增强字段是下游硬性契约, 则应 Fail-fast。

## 4. 通用工程经验

- 每个依赖先定义“必要”还是“增强”;
- 降级必须可观察、可统计、可告警;
- 不捕获无法正确处理的异常;
- 保留原始顺序和原始数据是安全回退;
- 为失败组合写表驱动测试, 不只测成功路径。

## 5. 实践练习

实现并行聚合器, 同时调用天气源 A 和 B:

```text
WeatherResult(
  data,
  sources_succeeded,
  sources_failed,
  degraded,
  reasons
)
```

要求:

1. 两路成功时合并;
2. 一路失败时返回另一路并标记降级;
3. 两路失败时抛出领域异常;
4. 成功但返回空数据不得记为异常;
5. 增加单路超时测试;
6. 写完四格失败矩阵测试。

## 6. 验收问题

1. 优雅降级与捕获所有异常有什么本质区别?
2. 为什么 Reranker 通常可以回退, Embedding 初始化未必可以?
3. 空结果和异常为什么必须分开?
4. `fallback_reason` 应服务哪些消费者?
5. 只实现异常捕获, 没有超时, 还缺少什么保障?

## 7. 今日学习成果

完成本课后, 你应能按业务必要性划定故障边界, 设计具有部分结果和显式状态的回退机制, 并用失败矩阵验证行为。
