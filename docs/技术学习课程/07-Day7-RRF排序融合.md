# Day 7: RRF 排序融合

## 今日定位

不同检索器的分数不在同一量纲。RRF 不比较原始分值, 只利用名次完成融合, 是混合检索中简单、稳健且可解释的排序方法。

## 学习目标

- 理解分数融合与排名融合的区别;
- 掌握 RRF 公式和参数 $k$;
- 理解重复候选、权重和确定性排序;
- 能独立实现并测试 RRF。

## 1. 技术背景

Dense 结果可能使用余弦相似度, BM25 使用词频统计后的非归一化分数。直接相加存在问题:

```text
Dense: 0.82, 0.79, 0.76
BM25: 18.4, 12.1, 6.3
```

BM25 会因数值更大支配结果, 但这不表示它更可靠。即使做 Min-Max 归一化, 分布和异常值也会影响结果。

## 2. 核心原理

对于候选文档 $d$, RRF 分数为:

$$
RRF(d)=\sum_{r\in R}\frac{1}{k+\operatorname{rank}_r(d)}
$$

- $R$ 是排名列表集合;
- rank 从 1 开始;
- $k$ 控制头部名次差异的影响, 项目默认 60。

同一候选被多路召回时会累计贡献。因此“两路都认为相关”的候选自然获得更高分。

### 2.1 参数 $k$

- $k$ 小: 更强调第一名和后续名次的差异;
- $k$ 大: 各名次贡献更平滑。

$k$ 不是精确率开关, 应通过 Golden Set 验证。

### 2.2 Weighted RRF

带权版本:

$$
RRF_w(d)=\sum_{r\in R}
\frac{w_r}{k+\operatorname{rank}_r(d)}
$$

当某一路在特定领域长期更可靠时可以赋更高权重, 但权重会降低算法“免校准”的优势。

### 2.3 确定性

两个候选 RRF 分数相同时, 必须指定稳定的第二排序键。否则底层容器顺序或并发完成顺序可能导致结果漂移, 使测试和缓存不稳定。

## 3. 项目案例分析

### 3.1 阅读文件

- `src/core/query_engine/fusion.py`;
- `tests/unit/test_fusion_rrf.py`;
- `src/core/query_engine/hybrid_search.py` 中 `_fuse_results()`。

### 3.2 `RRFFusion.fuse()` 的步骤

```text
ranking_lists
  → 逐列表枚举 rank
  → 以 chunk_id 聚合 1 / (k + rank)
  → 使用首次出现结果保存文本与元数据
  → 转换为统一 RetrievalResult
  → 按 (-score, chunk_id) 排序
  → 截断 top_k
```

`chunk_id` 是跨列表识别同一候选的主键。如果 Dense 和 Sparse 对同一内容使用不同 ID, RRF 无法累计贡献。

### 3.3 项目中的确定性设计

实现明确使用:

```text
(-rrf_score, chunk_id)
```

作为排序键。分数降序, ID 升序。这一细节保证相同输入产生相同输出。

### 3.4 回退边界

当没有 Fusion 组件时, `HybridSearch` 采用交错合并并按 `chunk_id` 去重。这是可用性回退, 不等于 RRF, 质量行为也不同。

## 4. 通用工程经验

- 排名列表中的 rank 必须从 1 开始并保持定义一致;
- 去重主键应稳定且跨召回器统一;
- 对空列表、单列表、重复项和并列分数写测试;
- 保留来源 rank, 便于解释融合结果;
- Weighted RRF 的权重必须来源于评估数据, 不能凭感觉调整。

## 5. 实践练习

给定:

```text
Dense: A, B, C, D
BM25:  B, E, A, F
```

任务:

1. 用 $k=60$ 手算 A、B、C、E 的 RRF 分数;
2. 实现 `fuse(rankings, k, top_k)`;
3. 增加同分时按 ID 排序;
4. 实现 Weighted RRF;
5. 比较权重 `[1,1]` 与 `[2,1]` 的排序。

必须覆盖这些测试:

- 输入为空;
- 只有一个列表;
- 一个候选在同一列表重复出现;
- 多个候选同分;
- `top_k` 大于候选总数。

4. 为什么 `chunk_id` 是融合正确性的前提?
