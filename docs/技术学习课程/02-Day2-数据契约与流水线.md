# Day 2: 数据契约与流水线设计

## 今日定位

Day 1 建立了 RAG 系统的整体心智模型。Day 2 开始向系统内部深入, 但今天仍然不学具体检索算法。

今天要解决的是一个更基础的工程问题:

> 当一份文档需要经过解析、分块、增强、编码和存储等多个阶段时, 如何保证数据在阶段之间正确流动, 并且让每个组件可以独立理解、替换、测试和排错?

这个问题的答案不是“写一个很长的函数”, 而是:

- 定义稳定的数据契约;
- 把处理过程分解为职责明确的阶段;
- 明确每个阶段必须保持的不变量;
- 由流水线编排器组织这些阶段;
- 让错误和状态成为契约的一部分。

## 今日学习目标

学完后, 你应该能够:

1. 区分数据结构、数据契约和存储 Schema。
2. 说明为什么多阶段系统需要稳定的中间表示。
3. 理解 Pipeline 模式中“阶段”和“编排”的职责边界。
4. 找出跨阶段必须保持的数据不变量。
5. 判断一个字段应该是显式属性, 还是放在可扩展元数据中。
6. 从源码中识别“文档声称的契约”和“代码真正强制的契约”之间的差异。
7. 为一个脱离 RAG 项目的数据处理任务设计最小流水线。

## 第一部分: 什么是数据契约

## 1. 从函数签名理解契约

一个简单函数可以写成:

```python
def split(text: str) -> list[str]:
    ...
```

这个签名已经表达了一部分契约:

- 输入应该是字符串;
- 输出应该是字符串列表。

但它没有说明:

- `text` 能不能是空字符串;
- 输出列表能不能为空;
- 输出是否保持原文顺序;
- 是否允许丢失文本;
- 分块是否需要保留来源;
- 发生错误时是报错、返回空列表, 还是返回原文。

因此, 数据契约不只是类型标注。

一份完整契约至少需要表达:

```text
数据形状
+ 必填字段
+ 字段语义
+ 取值约束
+ 跨字段关系
+ 顺序与数量不变量
+ 错误语义
+ 版本演进方式
```

## 2. 数据结构、数据契约和 Schema 的区别

### 数据结构

数据结构描述数据在程序中如何组织。

例如:

```python
{"id": "c1", "text": "...", "metadata": {}}
```

它告诉我们有哪些键, 却不一定告诉我们字段的业务意义。

### 数据契约

数据契约描述组件之间对数据的共同承诺。

例如:

- `id` 必须稳定且唯一;
- `text` 必须是对应原文的非空文本;
- `metadata.source_path` 必须存在;
- `chunk_index` 必须表示源文档中的顺序;
- 输出 Chunk 顺序必须与原文顺序一致。

### 存储 Schema

Schema 通常是数据在特定存储或交换系统中的物理表达。

例如:

- SQLite 表的列;
- JSON Schema;
- Protobuf Message;
- 向量库中的 ID、Vector 和 Metadata;
- 消息队列中的 Event Schema。

三者的关系可以理解为:

```text
业务语义与约束 = 数据契约
              ↓
程序内的表达 = 数据结构
              ↓
存储/传输中的表达 = Schema
```

## 3. 契约不是文档, 而是可被执行的约束

只在注释中写“`source_path` 必填”, 不代表系统真正强制了它。

契约可以通过多个层次执行:

1. 类型检查: 开发期发现类型不匹配。
2. 构造时校验: 对象一创建就检查必填字段。
3. 边界校验: 进入一个阶段前检查输入。
4. 存储约束: 唯一键、非空列、外键或 Schema Validator。
5. 契约测试: 验证所有实现都符合同一行为。

优秀的契约会尽量在错误刚产生时拦截, 而不是让错误数据流到三个阶段之后才爆炸。

## 第二部分: 为什么需要中间表示

## 1. 直接传递字典的问题

如果所有阶段都传递一个自由字典:

```python
item = {
    "content": "...",
    "source": "...",
}
```

下一个阶段可能将字段改成:

```python
item = {
    "text": "...",
    "source_path": "...",
}
```

再下一个阶段又假设存在:

```python
item["metadata"]["path"]
```

这类问题往往不会在代码编译或对象创建时出现, 而会在运行到特定数据时以 `KeyError` 的形式爆发。

典型后果:

- 字段命名漂移;
- 元数据悄悄丢失;
- 下游依赖上游的实现细节;
- 只能通过长链路端到端测试发现问题;
- 一个阶段更改字段后, 多个下游模块同时损坏。

## 2. 中间表示的作用

中间表示不是单纯“换一个类名”。它的真正作用是隔离变化。

```text
PDF Loader 的输出细节
          ↓
       Document
          ↓
Splitter 不需要知道 PDF SDK
          ↓
        Chunk
          ↓
Encoder 不需要知道分块算法
          ↓
     ChunkRecord
          ↓
VectorStore 不需要知道文档解析过程
```

中间表示带来三个核心价值:

### 隔离上游实现

无论文档来自 PDF、Markdown 还是网页, 只要最终生成符合契约的 `Document`, 下游就不需要知道文档来源。

### 隔离下游存储

上游处理组件产生 `ChunkRecord`, 不需要知道数据最终存在 Chroma、Qdrant 还是其他数据库。

### 建立可测试边界

每个阶段都可以用人工构造的中间对象独立测试, 不必真正解析 PDF 或连接向量数据库。

## 3. 粗粒度与细粒度表示

中间表示并非越多越好。

如果为每一个小操作都定义一个类, 系统会充满只使用一次的转换对象。

判断是否需要新中间表示, 可以问:

1. 数据的语义是否发生了明显变化?
2. 是否进入了新的生命周期阶段?
3. 是否多个组件需要共享该数据形状?
4. 是否需要在该边界进行独立校验、序列化或测试?

如果答案大部分为“是”, 定义新的中间表示才有明确价值。

## 第三部分: 当前项目的核心数据契约

## 1. Document: 规范化文档

源码位置:

```text
src/core/types.py
```

核心字段:

```python
@dataclass
class Document:
    id: str
    text: str
    metadata: dict[str, Any]
```

它的语义是:

- 已经从外部文件或数据源中读取;
- 已转换为系统统一理解的文本形式;
- 尚未拆分成可检索的 Chunk;
- 仍保留文档级来源和图片信息。

项目强制的契约:

```python
if "source_path" not in self.metadata:
    raise ValueError(...)
```

这意味着 `source_path` 不是普通扩展字段, 而是完成来源追溯的必要字段。

## 2. Chunk: 可独立处理的文本单元

核心字段:

```python
@dataclass
class Chunk:
    id: str
    text: str
    metadata: dict[str, Any]
    start_offset: int | None
    end_offset: int | None
    source_ref: str | None
```

相比 `Document`, `Chunk` 增加了三类语义:

### 位置

- `chunk_index`;
- `start_offset`;
- `end_offset`。

这些字段回答“它在原文什么位置”。

### 归属

- `source_ref`;
- `source_path`。

这些字段回答“它从哪份文档来”。

### 可独立处理

Chunk 已经是可被增强、编码、存储和检索的基本单元。

## 3. ChunkRecord: 可存储的检索记录

核心字段:

```python
@dataclass
class ChunkRecord:
    id: str
    text: str
    metadata: dict[str, Any]
    dense_vector: list[float] | None
    sparse_vector: dict[str, float] | None
```

它不再只是文本片段, 而是“已完成检索表示、准备写入存储”的记录。

`from_chunk` 通过复制元数据生成新对象:

```python
metadata=chunk.metadata.copy()
```

这个细节的意义是: 新记录不与原 Chunk 共享同一个顶层元数据字典, 减少无意中修改上游对象的风险。

## 4. ProcessedQuery: 查询侧中间表示

```python
@dataclass
class ProcessedQuery:
    original_query: str
    keywords: list[str]
    filters: dict[str, Any]
    expanded_terms: list[str]
```

它将用户输入与下游检索器隔离:

- Dense Retriever 可以使用 `original_query`;
- Sparse Retriever 可以使用 `keywords`;
- 两者都可以使用 `filters`;
- 未来查询扩展可以使用 `expanded_terms`。

## 5. RetrievalResult: 隔离不同检索实现

```python
@dataclass
class RetrievalResult:
    chunk_id: str
    score: float
    text: str
    metadata: dict[str, Any]
```

Dense 和 Sparse 的内部计分方法完全不同, 但融合层不需要处理两种后端的原始返回结构。它们先被转换成统一的 `RetrievalResult`。

这就是契约带来的解耦:

```text
Dense backend response ─→ RetrievalResult ─┐
                                        ├→ Fusion / Rerank / Response
Sparse backend response → RetrievalResult ─┘
```

## 第四部分: 显式字段与可扩展 Metadata

## 1. 为什么不把所有元数据定义成属性

不同文档类型的元数据差异很大:

- PDF 有页码、图片坐标和页数;
- Markdown 有标题层级;
- 网页有 URL、抓取时间和 DOM 路径;
- 财报有表格、报告期和公司简称。

如果把所有可能字段都写成显式属性, 数据类会迅速膨胀, 并且大量字段在大多数场景下都是 `None`。

所以项目采用:

```text
稳定核心字段 + 可扩展 metadata
```

## 2. 哪些字段应该显式存在

一个字段值得成为显式属性, 通常满足一个或多个条件:

- 所有实例都需要;
- 决定对象身份;
- 被多个下游阶段直接依赖;
- 需要强类型或构造时校验;
- 对序列化、排序或数据一致性很重要。

例如 `Chunk.text`、`Chunk.id`、`RetrievalResult.score`。

## 3. 哪些字段适合放在 Metadata

适合放入 Metadata 的字段通常:

- 只适用于某些文档类型;
- 属于可选增强结果;
- 下游可以在缺失时继续运行;
- 需要快速增加, 但不值得修改核心契约。

例如:

- `page`;
- `title`;
- `summary`;
- `tags`;
- `image_captions`;
- `refined_by`;
- `enriched_by`。

## 4. 自由 Metadata 的代价

`dict[str, Any]` 很灵活, 但它将问题从“修改数据类”转移成“所有人必须记住字段约定”。

典型风险:

- `page`、`page_num`和 `page_number` 并存;
- 同一字段有时是整数, 有时是字符串;
- 一个 Transform 直接修改共享字典, 影响其他阶段;
- 字段在上游丢失后, 直到引用生成时才被发现;
- 类型检查工具难以帮助发现问题。

通用改善思路不是“禁止 Metadata”, 而是:

- 建立字段命名表;
- 区分必填与可选字段;
- 在边界进行校验;
- 对高频字段使用 `TypedDict`、Pydantic 或专用数据类;
- 用契约测试检查不同实现。

## 第五部分: Pipeline 模式

## 1. Pipeline 不是“按顺序调函数”这么简单

一条真正的工程流水线通常需要同时解决:

- 阶段顺序;
- 数据转换;
- 进度报告;
- 失败策略;
- 幂等性;
- 中间状态;
- 观测信息;
- 资源释放;
- 最终结果摘要。

因此 Pipeline 的价值是“编排”, 不是亲自实现每个算法。

## 2. 项目的六阶段摄取流水线

源码位置:

```text
src/ingestion/pipeline.py
```

数据流:

```text
文件路径
   ↓
Stage 1: Integrity
   输出: file_hash + 是否跳过
   ↓
Stage 2: Load
   输出: Document
   ↓
Stage 3: Split
   输出: Chunk[]
   ↓
Stage 4: Transform
   输出: 增强后 Chunk[]
   ↓
Stage 5: Encode
   输出: Dense Vectors[] + Sparse Stats[]
   ↓
Stage 6: Store
   输出: Vector IDs[] + 索引状态
   ↓
PipelineResult
```

## 3. 编排器应该做什么

当前 `IngestionPipeline` 负责:

- 根据配置初始化组件;
- 决定后端组合;
- 按正确顺序调用阶段;
- 把上一阶段输出交给下一阶段;
- 记录进度和耗时;
- 汇总每个阶段的结果;
- 在全部完成后标记成功;
- 发生错误时返回失败结果。

## 4. 编排器不应该做什么

理想情况下, Pipeline 不应该亲自:

- 解析 PDF 字节;
- 实现分块算法;
- 计算 Embedding;
- 实现 BM25 公式;
- 处理向量库 SDK 的细节;
- 生成图片描述。

这些应该由专门组件处理。Pipeline 只需要知道它们接收什么、返回什么、失败时是什么语义。

## 5. 阶段结果与总结果

`PipelineResult` 保存:

- 整体是否成功;
- 文件路径;
- 文档 ID;
- Chunk 数量;
- 图片数量;
- 向量 ID;
- 错误信息;
- 每个阶段的结果摘要。

这种返回方式比只返回 `True/False` 更有价值, 因为调用者可以区分:

- 完整成功;
- 因已处理而跳过;
- 在某个阶段失败;
- 完成了多少 Chunk;
- 写入了哪些向量记录。

## 第六部分: 跨阶段不变量

不变量是指在某个范围内必须始终成立的条件。

多阶段系统最容易出现的问题, 不是单个函数算错, 而是一个阶段破坏了下游假设。

## 1. 数量对齐

在正常成功路径上:

```text
len(chunks)
= len(dense_vectors)
= len(sparse_stats)
= len(vector_ids)
```

一旦这个关系被破坏, 即使每个列表自身都是合法的, 整体数据也已经错位。

## 2. 顺序对齐

不仅数量要相同, 位置也必须对应:

```text
chunks[i] ↔ dense_vectors[i] ↔ sparse_stats[i] ↔ vector_ids[i]
```

如果编码阶段为了并行处理而改变了结果顺序, 又没有使用 Chunk ID 重新关联, 就可能把 A 文本的向量存到 B 文本上。

这是一种很危险的错误: 系统可以正常运行, 但检索质量会静默变差。

## 3. 身份稳定

同一个逻辑 Chunk 在重复运行时应该获得稳定 ID, 否则 Upsert 会变成不断新增记录。

项目在 `VectorUpserter` 中使用:

```text
source_path hash + chunk_index + content hash
```

构建向量记录 ID。

## 4. 来源可追溯

从 `Document` 到 `Chunk`, 再到存储记录, 以下信息不应丢失:

- 源文件;
- 父文档 ID;
- Chunk 位置;
- 页码或章节;
- 文档哈希;
- 图片引用。

只要其中一项在中间阶段被覆盖, 后续的 Citation 就无法凭空恢复它。

## 5. 成功标记必须最后写入

项目在向量、BM25 和图片索引全部处理完成后, 才调用 `mark_success`。

如果在分块完成时就标记成功, 但后续向量写入失败, 下次运行可能因“文件已处理”直接跳过, 从而把半成品当成完整数据。

## 6. 同一集合的多索引必须使用同一身份

项目在写入 BM25 前, 把向量库产生的 ID 同步到 Sparse Stats:

```python
for stat, vid in zip(sparse_stats, vector_ids):
    stat["chunk_id"] = vid
```

这样 Dense 和 Sparse 召回才能通过同一 `chunk_id` 识别“这是同一个 Chunk”, RRF 才能正确累加排名信号。

## 第七部分: 错误语义也是契约

## 1. 三种不同的失败语义

### 输入无效

例如:

- Chunk 列表为空;
- Chunk 数量与 Vector 数量不一致;
- 缺少 `source_path`;
- `batch_size <= 0`。

这类问题应该立即失败, 因为继续运行不会产生有意义结果。

### 可降级的增强失败

例如:

- LLM Chunk Refinement 失败, 但原 Chunk 仍然可用;
- Metadata Enrichment 失败, 但可以保留基础元数据;
- 图片描述失败, 但文本仍可以索引。

此时可以返回原对象, 但应该记录已降级。

### 破坏整体一致性的失败

例如:

- 一部分 Chunk 有 Vector, 另一部分没有;
- Dense 已写入, BM25 完全失败;
- 存储的 Chunk ID 与索引 ID 不同。

这类问题不应该被当作普通降级。因为系统可能返回成功, 但已经产生难以发现的数据不一致。

## 2. 结果对象比布尔值更有信息量

与其只返回:

```python
return True
```

更有价值的是:

```python
PipelineResult(
    success=True,
    doc_id="...",
    chunk_count=20,
    vector_ids=[...],
    stages={...},
)
```

这样调用者可以决定:

- 是否可以进入下一个业务流程;
- 是否需要重试;
- 哪个阶段需要排查;
- 是否完成了预期数量的处理;
- 是否只是因内容未变而跳过。

## 第八部分: 项目源码深度观察

本节不是为了批评代码, 而是练习一个重要能力:

> 不仅阅读类名和注释, 还要验证实现是否真正保持了声称的契约。

## 观察 1: 核心字段只被部分强制

`Document`、`Chunk`、`ChunkRecord` 都强制 `metadata.source_path` 存在。

但某些文档中声称必要的字段, 例如 `chunk_index`, 并没有在 `Chunk.__post_init__` 中统一强制, 而是到 `VectorUpserter._generate_chunk_id` 时才校验。

这意味着:

- 一个缺少 `chunk_index` 的 Chunk 仍能被构造;
- 它可能已经通过了多个阶段;
- 直到存储前才失败。

这是“错误发现位置晚于错误产生位置”的典型现象。

## 观察 2: 顺序契约是隐式的

`BatchProcessor` 返回两个列表:

- `dense_vectors`;
- `sparse_stats`。

调用者依赖它们与原 `chunks` 顺序一致, 但这一关系只体现在注释和实现方式中, 没有通过 `chunk_id` 显式关联。

这种契约称为位置契约。

位置契约很高效, 但容易在并发、过滤或部分失败时被破坏。

## 观察 3: 部分批次失败可能破坏数量对齐

`BatchProcessor.process` 在某个批次失败后:

- 记录 `failed_chunks`;
- 不为该批次追加 Dense 和 Sparse 结果;
- 继续处理后续批次。

但 Pipeline 后续仍使用原始 `chunks` 列表与 `dense_vectors` 写入。

如果真的出现部分批次失败, 可能出现:

```text
len(chunks) != len(dense_vectors)
```

`VectorUpserter` 会在写入前拒绝该数据。这防止了错位写入, 但也说明“批次失败后继续”与“整批 Chunk 一起写入”的语义并未完全对齐。

这里的通用经验是:

> 允许局部失败时, 结果不能只返回“成功值列表”, 还必须保留成功值与原输入的显式关联。

例如返回:

```python
EncodedChunk(
    chunk_id="c1",
    dense_vector=[...],
    sparse_stats={...},
    error=None,
)
```

而不是只返回两个依赖位置对齐的列表。

## 观察 4: 设计说明与实际执行需要分开验证

`BatchProcessor` 的设计说明中写有“Dense 和 Sparse 独立并行”, 但当前代码是先调用 Dense Encoder, 再调用 Sparse Encoder。

这个例子提醒我们:

- 注释表达的可能是设计意图;
- 调用关系表达的才是当前真实行为;
- 阅读源码时不能用注释代替对执行路径的确认。

## 观察 5: 可观测数据也有契约

Pipeline 为 `load`、`split`、`transform`、`embed`、`upsert` 等阶段记录 Trace。

如果 Dashboard 假设每个 Stage 都包含 `method`、`elapsed_ms`和 `chunks`, 那么 Trace 数据本身也是一份跨组件契约。

很多系统只给业务数据定义 Schema, 却忽略日志、Trace 和 Metric 的 Schema, 最终导致监控面板随着业务代码变化而失效。

## 第九部分: 通用工程经验

## 经验 1: 边界处严格, 组件内部灵活

在数据刚进入系统或刚离开一个阶段时严格校验, 可以让组件内部不必在每一行都写防御性代码。

```text
外部不可信输入
        ↓ 严格校验
内部可信数据契约
        ↓
简单的业务逻辑
```

## 经验 2: 重要关系不要只靠列表位置表达

当系统可能出现以下情况时, 应考虑用 ID 显式关联:

- 并发处理;
- 过滤;
- 重试;
- 部分失败;
- 批次乱序;
- 跨进程传输。

## 经验 3: 每个阶段只完成一次语义跃迁

好的阶段输入和输出之间有清晰含义:

```text
原始文件 → 规范化文档
规范化文档 → 可检索片段
可检索片段 → 增强片段
增强片段 → 数学表示
数学表示 → 持久化记录
```

如果一个阶段同时解析 PDF、分块、调 LLM、计算向量并写数据库, 那么它实际上不是一个阶段, 而是一条被隐藏在函数中的流水线。

## 经验 4: 错误信息应该指向契约违反

好的错误:

```text
Chunk count (10) must match vector count (9)
```

差的错误:

```text
Processing failed
```

前者告诉你被破坏的是“数量对齐契约”, 后者只告诉你某处出错。

## 经验 5: 流水线完成不等于所有增强都成功

流水线结果最好区分:

- 核心任务是否完成;
- 哪些可选增强失败;
- 是否发生降级;
- 最终数据是否仍满足存储和检索契约。

## 第十一部分: 参考答案

## 练习 1 参考实现

```python
from dataclasses import dataclass, field
from datetime import datetime


@dataclass(frozen=True)
class RawText:
    source: str
    content: str
    loaded_at: datetime


@dataclass(frozen=True)
class CleanDocument:
    document_id: str
    source: str
    text: str

    def __post_init__(self) -> None:
        if not self.text.strip():
            raise ValueError("document text cannot be empty")


@dataclass(frozen=True)
class Paragraph:
    paragraph_id: str
    document_id: str
    index: int
    text: str

    def __post_init__(self) -> None:
        if self.index < 0:
            raise ValueError("paragraph index must be non-negative")
        if not self.text.strip():
            raise ValueError("paragraph text cannot be empty")


@dataclass(frozen=True)
class EnrichedParagraph:
    paragraph: Paragraph
    keywords: tuple[str, ...] = field(default_factory=tuple)
    summary: str = ""
```

这个实现刻意让 `EnrichedParagraph` 持有原 `Paragraph`, 而不是重新复制和修改原文字段。

## 练习 2 参考不变量

1. `source` 在全链路不能丢失。
2. 同一文档在内容不变时应生成同一 `document_id`。
3. `Paragraph.document_id` 必须指向它的父文档。
4. Paragraph 顺序必须与原文顺序一致。
5. `index` 必须从0开始且不重复。
6. Enrich 不能修改 Paragraph 原文。
7. 成功的增强结果必须显式关联 `paragraph_id`。
8. Save 返回的 ID 必须与待写入记录一一对应。
9. 只有所有必要数据写入成功后才能标记文档完成。
10. 重复处理同一记录不能产生重复存储数据。

## 练习 3 参考答案

### 被破坏的契约

`paragraphs[i]` 与 `summaries[i]` 一一对应的位置契约被破坏。

### 为什么不会报错

`zip` 只会遍历较短列表的长度, 不会因长度不同自动报错。

### 错位方式

如果第2个 Paragraph 生成摘要失败, `summaries[1]` 实际属于第3个 Paragraph, 但会被写入第2个 Paragraph。

### 更安全的设计

```python
results = []
for paragraph in paragraphs:
    try:
        summary = summarize(paragraph.text)
        results.append((paragraph.paragraph_id, summary, None))
    except Exception as exc:
        results.append((paragraph.paragraph_id, None, str(exc)))
```

关键是每个结果显式保留 `paragraph_id`, 并且失败也占据一个可追踪的结果位置。

## 练习 4 参考答案

1. 空文档: 立即失败, 因为没有可处理的核心数据。
2. 单个摘要超时: 可重试一次; 仍失败时使用原 Paragraph 降级, 并记录失败状态。
3. 全部摘要超时: 如摘要是可选增强, 整体降级为无摘要模式; 如摘要是核心产物, 则失败。
4. 数据库临时断开: 有界限地重试; 超出重试预算后整体失败, 不写成功标记。
5. 缺少 `document_id`: 立即拒绝该数据; 这是身份契约违反, 不能用降级补救。
6. 返回 ID 少一个: 整体失败或进入补偿流程; 这表明写入数量契约已破坏。

## 练习 5 参考方向

| 边界 | 输入 | 输出 | 显式校验 | 隐式假设 |
|---|---|---|---|---|
| Loader → Chunker | `Document` | `Chunk[]` | Document有非空文本与`source_path` | 文本中图片占位符与Metadata一致 |
| Chunker → Transform | `Chunk[]` | 增强后`Chunk[]` | Chunk有`source_path` | 数量、顺序、ID和原文语义保持 |
| Transform → Encoder | `Chunk[]` | Vector/Stats | 列表非空 | 每个Chunk都有可编码文本 |
| Encoder → Storage | Chunk、Vector、Stats | IDs | Chunk与Vector数量相等 | 顺序一一对应, Sparse也与同一Chunk对应 |
