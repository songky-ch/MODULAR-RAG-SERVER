# Day 9: 可插拔架构与依赖倒置

## 今日定位

可插拔架构的目标不是“类越多越高级”, 而是让核心业务依赖稳定能力契约, 把模型供应商、数据库和算法实现等变化隔离在边界之外。

## 学习目标

- 理解依赖倒置、Strategy、Factory 和 Adapter;
- 能区分稳定职责与易变实现;
- 看懂项目中 Base 接口、Factory 与核心编排的协作;
- 避免无价值的抽象和工厂泛滥。

## 1. 技术背景

AI 基础设施变化频繁:

- Embedding Provider 可能替换;
- 向量库可能从 Chroma 切换到其他后端;
- 分块策略随文档类型变化;
- Reranker 可能在本地模型、API 和 LLM 间切换。

如果核心流水线直接调用某个 SDK, 替换后端会扩散到整个系统。依赖倒置要求高层策略依赖抽象, 具体实现也遵循该抽象。

## 2. 核心原理

### 2.1 四个角色

```text
Core Policy → Stable Interface ← Adapter → External SDK
                    ↑
                 Factory
                    ↑
                 Config
```

- Stable Interface: 定义系统真正需要的最小能力;
- Adapter: 把外部 SDK 的参数和返回值转换成内部契约;
- Factory: 根据配置在组合根创建具体实现;
- Core Policy: 只编排能力, 不感知供应商细节。

### 2.2 Strategy 与 Factory

Strategy 解决“运行时使用哪种行为”; Factory 解决“如何创建该行为对象”。二者职责不同:

- `BaseSplitter` 的不同实现是 Strategy;
- `SplitterFactory` 根据配置构造 Strategy。

### 2.3 依赖注入

构造函数注入让依赖显式化:

```text
HybridSearch(dense_retriever, sparse_retriever, fusion)
```

它比在方法内部创建全局对象更易测试, 也能替换为 Fake 或 Stub。

### 2.4 抽象的成立条件

只有存在真实变化轴, 且调用方需要统一语义时才值得抽象。若只有一个实现且没有明确变化需求, 额外 Base 和 Factory 可能只是间接层。

## 3. 项目案例分析

### 3.1 重点抽象族

| 能力 | 抽象 | 创建入口 |
|---|---|---|
| Embedding | `BaseEmbedding` | `EmbeddingFactory` |
| Splitter | `BaseSplitter` | `SplitterFactory` |
| Vector Store | `BaseVectorStore` | Vector Store Factory |
| Reranker | `BaseReranker` | `RerankerFactory` |
| Evaluator | `BaseEvaluator` | Evaluator Factory/Composite |
| LLM | Base LLM 契约 | LLM Factory |

对应目录位于 `src/libs/embedding/`、`src/libs/splitter/`、`src/libs/vector_store/`、`src/libs/reranker/`、`src/libs/evaluator/` 和 `src/libs/llm/`。

### 3.2 Embedding 调用链

```text
Settings
  → EmbeddingFactory 创建 Provider Adapter
  → DenseEncoder 依赖 BaseEmbedding
  → Pipeline 只调用统一 embed_documents
```

Provider 的鉴权、请求字段、限流和返回解析留在 Adapter 内, 不泄漏给 Pipeline。

### 3.3 Vector Store 契约

`BaseVectorStore` 定义 Upsert、Query 等核心能力。具体 Chroma 实现把领域记录转换为后端格式。调用方不应直接依赖 Chroma Collection 对象。

### 3.4 核心层二次编排

`CoreReranker` 并非另一个模型实现。它在 `BaseReranker` 之上补充配置、对象转换、Trace 和回退。这体现:

- `libs` 封装技术能力;
- `core` 表达业务执行策略。

## 4. 通用工程经验

- 接口应围绕调用方需求设计, 不是完整复制第三方 SDK;
- 外部类型止步于 Adapter, 核心层使用自己的领域对象;
- Factory 应集中在组合根, 不要到处隐藏对象创建;
- 配置只决定策略选择, 不应承载复杂业务判断;
- 契约测试应对所有实现重复运行, 确保可替换性真实成立。

## 5. 实践练习

实现一个可插拔搜索器:

```text
SearchBackend.search(query) → SearchResult[]
```

提供两个实现:

- `MemorySearchBackend`;
- `JsonFileSearchBackend`。

要求:

1. 业务层只依赖 `SearchBackend`;
2. Factory 根据配置创建实现;
3. 外部 JSON 字段只在 Adapter 内转换;
4. 为两个实现运行同一组契约测试;
5. 用 Fake Backend 测试业务层, 不访问真实文件。

最后删除一个没有真实用途的抽象层, 说明为什么更简单的设计反而更清晰。

## 6. 验收问题

1. 依赖倒置与依赖注入有什么区别?
2. Strategy 和 Factory 分别解决什么问题?
3. 为什么不能让第三方 SDK 类型穿透核心层?
4. 契约测试如何证明实现可替换?
5. 什么情况下不应该创建抽象接口?

## 7. 今日学习成果

完成本课后, 你应能识别系统中的变化点, 用最小稳定接口隔离变化, 并判断一层抽象究竟降低还是增加复杂度。
